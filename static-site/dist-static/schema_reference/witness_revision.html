Witness Revision

A Witness Revision provides cryptographic proof that a revision existed at a specific point in time. It anchors revision hashes to external systems like blockchains (Ethereum), decentralized networks (Nostr), or Trusted Timestamping Authorities (TSA). This creates an immutable, verifiable timestamp that cannot be backdated.

Overview

Witness revisions provide:
    1. Timestamping: Cryptographic proof of when a revision existed
    1. Immutability: Blockchain or TSA anchoring prevents backdating
    1. Verifiability: Anyone can verify the witness independently
    1. Batch Efficiency: Multiple revisions can be witnessed in a single transaction via Merkle trees
    1. Decentralization: Uses public blockchains or decentralized networks

Schema Structure

Common Fields

| Field | Type | Required | Description | |-------|------|----------|-------------| | previous_revision | string | Yes | Hash reference to the revision being witnessed | | revision_type | string | Yes | Always "witness" for witness revisions | | nonce | string | Yes | Random 16-byte hex string for uniqueness | | local_timestamp | number | Yes | Unix timestamp when the witness was created locally | | version | string | Yes | Protocol version: "https://aqua-protocol.org/docs/v4/schema" | | method | string | Yes | Canonicalization method: "scalar" (typical) or "tree" | | hash_type | string | Yes | Hash algorithm: "FIPS_202-SHA3-256" | | witness | object | Yes | Witness value object containing proof details |

Witness Value Object

The witness field contains details about the witnessing transaction: | Field | Type | Required | Description | |-------|------|----------|-------------| | merkle_proof | array | Yes | Array of hex strings forming the Merkle proof path | | sender_account_address | string | Yes | Address that submitted the witness transaction | | transaction_hash | string | Yes | Transaction hash on the blockchain or network | | smart_contract_address | string | Yes | Address of the witness contract (or endpoint URL) | | network | string | Yes | Network identifier: "mainnet", "sepolia", "holesky", "tsa", or "nostr" | | merkle_root | string | Yes | Root hash of the Merkle tree stored on-chain | | timestamp | number | Yes | Unix timestamp from the blockchain/network |

Network Types

1. Ethereum Networks

#### Mainnet
    1. Network: "mainnet"
    1. Purpose: Production witnessing on Ethereum mainnet
    1. Cost: Higher gas fees
    1. Security: Maximum security and permanence
#### Sepolia (Testnet)
    1. Network: "sepolia"
    1. Purpose: Testing and development
    1. Cost: Free testnet ETH
    1. Security: Testnet only, subject to resets
#### Holesky (Testnet)
    1. Network: "holesky"
    1. Purpose: Testing and development
    1. Cost: Free testnet ETH
    1. Security: Testnet only, newer Ethereum testnet

2. Trusted Timestamping Authority (TSA)

    1. Network: "tsa"
    1. Purpose: RFC 3161 compliant timestamping
    1. Provider: DigiCert or other TSA services
    1. Verification: Via TSA public certificates

3. Nostr

    1. Network: "nostr"
    1. Purpose: Decentralized social network timestamping
    1. Verification: Via Nostr relays and events
    1. Cost: Typically free

Merkle Tree Batching

To reduce costs, multiple revisions can be witnessed in a single transaction using Merkle trees.

Process

  1. Collect Revisions: Gather multiple revision hashes to witness
  1. Build Merkle Tree: Create a Merkle tree from the revision hashes
  1. Submit Root: Submit only the Merkle root to the blockchain
  1. Generate Proofs: For each revision, generate its Merkle proof
  1. Create Witnesses: Create witness revisions with proofs

Example: Witnessing 4 Revisions

` Revision Hashes:
    1. Rev1: 0xaaaa...
    1. Rev2: 0xbbbb...
    1. Rev3: 0xcccc...
    1. Rev4: 0xdddd...
Merkle Tree: Root / \ H12 H34 / \ / \ Rev1 Rev2 Rev3 Rev4 Submit to blockchain: Root hash only Witness for Rev1: merkle_proof: [H12's sibling (Rev2), H12's parent's sibling (H34)] merkle_root: Root `

Proof Verification

To verify a revision was witnessed:
  1. Start with revision hash
  1. Apply Merkle proof hashes (sibling hashes)
  1. Compute up the tree to get root
  1. Verify computed root matches merkle_root
  1. Verify merkle_root is on blockchain at transaction_hash

Validation Rules

A Witness Revision is valid if:
  1. Structure: Contains all required fields with correct types
  1. Revision Type: The revision_type is exactly "witness"
  1. Previous Revision: References a valid existing revision
  1. Network: One of the supported network values
  1. Timestamp Consistency:
- witness.timestamp should be close to local_timestamp - Both should be reasonable (not far in future)
  1. Merkle Proof: If present, must be valid array of hex strings
  1. Transaction Verification: The transaction_hash exists on the specified network
  1. Merkle Root Verification: If batched, Merkle proof must lead to merkle_root
  1. On-Chain Verification: The merkle_root is stored at the smart_contract_address in the transaction
  1. Address Format: All addresses must be valid hex strings

Witness Smart Contract

Ethereum Contract Interface

The witness contract typically has a method like: `solidity function witness(bytes32 merkleRoot) public returns (bool) `

Verification Process

  1. Query blockchain for transaction at transaction_hash
  1. Verify transaction called the smart_contract_address
  1. Extract the witnessed value (merkle_root) from transaction data
  1. Verify it matches the merkle_root in the witness revision
  1. Check transaction timestamp matches timestamp field

Common Use Cases

1. Document Notarization

` Document Object → Signature → Witness (Ethereum) ` Proves the document existed and was signed at a specific time.

2. Batch Processing

` 100 Documents → Compute Hashes → Build Merkle Tree → Single Witness Transaction ` Economical witnessing of many documents at once.

3. Legal Evidence

` Evidence Object → Signature (Parties) → Witness (Mainnet) ` Creates tamper-proof timestamp for legal proceedings.

4. Supply Chain

` Shipment Event → Witness (Every checkpoint) → Audit Trail ` Immutable record of supply chain events.

5. Credential Issuance

` VC Issuance → Signature (Issuer) → Witness (Public verification) ` Publicly verifiable credential issuance timestamps.

Timestamp Interpretation

local_timestamp

    1. Created by client when preparing witness revision
    1. Not trusted (can be manipulated)
    1. Used for user display and rough ordering

witness.timestamp

    1. Provided by blockchain/TSA
    1. Trusted (immutable on-chain)
    1. Authoritative time proof
    1. Should be used for verification and legal purposes
Best Practice: Always use witness.timestamp for critical timestamp verification.

Cost Considerations

Ethereum

| Network | Cost | When to Use | |---------|------|-------------| | Mainnet | ~$5-50 per transaction | Production, legal, high-value | | Sepolia | Free (testnet) | Development, testing | | Holesky | Free (testnet) | Development, testing | Gas Optimization: Batch multiple witnesses into one transaction using Merkle trees.

TSA

    1. Typically $0.10-1.00 per timestamp
    1. Good for compliance requirements
    1. RFC 3161 standard

Nostr

    1. Free (decentralized network)
    1. Lower trust than blockchain
    1. Good for social proof

Implementation Notes

Creating a Witness Revision

  1. Identify the revision(s) to witness
  1. Choose network (mainnet, sepolia, etc.)
  1. If batching, build Merkle tree
  1. Submit transaction to blockchain
  1. Wait for transaction confirmation
  1. Extract transaction details
  1. Generate Merkle proofs (if batched)
  1. Create witness revision(s)

Verifying a Witness Revision

#### Single Revision (No Batching) `
  1. Verify merkle_proof is empty
  1. Verify merkle_root equals previous_revision
  1. Query blockchain for transaction_hash
  1. Verify transaction exists and is confirmed
  1. Extract witnessed value from transaction
  1. Verify it matches merkle_root
  1. Check transaction timestamp
` #### Batched Revision `
  1. Verify merkle_proof is non-empty
  1. Compute Merkle root using proof and previous_revision
  1. Verify computed root matches merkle_root
  1. Query blockchain for transaction_hash
  1. Verify merkle_root is stored on-chain
  1. Validate transaction timestamp
`

Error Handling

Common issues and solutions: | Issue | Solution | |-------|----------| | Transaction not found | Wait longer (may not be confirmed yet) | | Wrong network | Check transaction on correct network | | Merkle proof invalid | Regenerate proof from original tree | | Timestamp mismatch | Check blockchain time vs local time | | Gas estimation failed | Check wallet balance and gas price |

Security Considerations

1. Network Selection

    1. Use mainnet for production/legal purposes
    1. Use testnets only for development
    1. Consider compliance requirements

2. Confirmation Depth

    1. Wait for sufficient confirmations (6+ for mainnet)
    1. More confirmations = higher security
    1. Balance security vs speed requirements

3. Merkle Proof Storage

    1. Store proofs securely
    1. Without proof, batched witness can't be verified
    1. Consider proof backup strategies

4. Timestamp Trust

    1. Trust blockchain timestamp over local_timestamp
    1. Account for block time variability
    1. Consider block reorganizations

5. Cost Management

    1. Batch witnesses to reduce costs
    1. Monitor gas prices
    1. Use appropriate network for use case

Advanced Topics

Multi-Network Witnessing

Witness the same revision on multiple networks for redundancy: ` Object → Signature → Witness (Ethereum) → Witness (Nostr) → Witness (TSA) ` Each witness provides independent proof.

Periodic Re-Witnessing

For long-term preservation, periodically re-witness important revisions: ` 2024: Object → Witness (Mainnet) 2025: Link to Object → Witness (Mainnet) 2026: Link to 2025 → Witness (Mainnet) `

Cross-Chain Verification

Witness on multiple blockchains for increased trust: ` Object → Witness (Ethereum Mainnet) → Witness (Polygon) → Witness (Arbitrum) ``

Relationship with Other Revisions

    1. Object Revisions: Primary target for witnessing
    1. Signature Revisions: Often witnessed after signing
    1. Link Revisions: Can be witnessed to prove connection timestamp
    1. Template Revisions: Rarely witnessed (they're standalone)
